# 产品路线图：从现状到满分

## P0 — 不做完不能叫 v1.0

| # | 事项 | 现状 | 目标 | 为什么必须 |
|---|------|------|------|-----------|
| 1 | 单元测试覆盖核心引擎 | 0 个单元测试 | executor/adapter/reviewer 各 ≥80% 覆盖率 | 14 个测试全是端到端。端到端慢且不隔离——改一行 executor 要跑 14 个测试才敢确认没炸。单元测试让你 1 秒知道 |
| 2 | Docker 沙箱 | ⚠️ 已标注 P0-1 未完成 | 代码节点在隔离容器里跑，不能碰宿主机文件系统 | 代码节点现在 spawn 子进程，安全边界是你的 Node 进程权限。不隔离，没法让别人用 |
| 3 | 一键体验工作流 | 用户打开是空白画布 | 启动后自动加载一个预置工作流（比如"输入一句话 → AI 翻译 → 输出"），点运行就能看到结果 | 新用户 30 秒的价值感知，不需要配 Key |

## P1 — 有竞争力必须做

| # | 事项 | 现状 | 目标 | ROI |
|---|------|------|------|-----|
| 4 | AI 对话驱动：增量修改 | 只能加节点/连线/运行 | 支持"把第 3 步改成 GPT-4"、"在步骤 2 和 3 之间插入一个翻译节点"、"这个节点跑失败了帮我修" | 核心差异化的厚度 |
| 5 | 执行结果可视化 | 日志窗口是纯文本 | 每个节点的输入/输出结构化展示，失败节点点开看堆栈，输出支持 JSON 树/表格/图片预览 | 调试体验从"看日志猜"变成"看数据判" |
| 6 | 前端 E2E 测试 (Playwright) | 49 张手动截图 | 10 个关键路径的自动化截图回归 | 改 UI 不再靠肉眼 diff 49 张图 |
| 7 | 路由拆分 | api.js 1552 行单体 | 拆成 routes/workflows.js / routes/models.js / routes/executions.js 等 | api.js 每加一个功能长 100 行，不拆就是定时炸弹 |

## P2 — 生态和体验深度

| # | 事项 | 现状 | 目标 |
|---|------|------|------|
| 8 | 工作流模板库 | TemplateModal 骨架存在 | 10 个内置模板（翻译/摘要/代码审查/数据清洗/文档生成/邮件报告等），一键加载 |
| 9 | 执行对比模式 | 无 | 同一个工作流跑两次，diff 输出差异——对调参场景极有价值 |
| 10 | CI/CD | 无 | GitHub Actions 自动跑全部测试 + lint + 构建，PR 合并前自动验证 |
| 11 | 错误恢复 | 节点失败整个流停 | 节点级重试 + 失败后跳过可选节点 + 部分成功结果仍然输出 |
| 12 | 文档站 | 只有 README + 使用说明.txt | API 文档自动生成 + 5 个带截图的场景教程 + 常见工作流模式参考 |

## 优先级路线图

```
现在 ──→ P0 (v0.9) ──→ P1 (v1.0) ──→ P2 (v1.5+)
        Docker沙箱       AI增量修改      模板库

        一键体验          E2E测试        CI/CD
                         路由拆分        错误恢复

```
